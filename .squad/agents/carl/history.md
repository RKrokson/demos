- **2025-07-27 (dns-resolution-diagnosis):** End-to-end DNS diagnosis for Cosmos DB 403 from Foundry BYO VNet. **ROOT CAUSE:** `enable_dns_link` defaults to `false` in Foundry-byoVnet. Without this toggle, the Foundry VNet has no custom DNS servers — it uses Azure default DNS (168.63.129.16), which checks private DNS zones linked to the querying VNet. The private DNS zones (created by AVM module in Networking) are only linked to the dns_vnet, NOT to the Foundry VNet. So Azure DNS returns Cosmos DB's public IP. Traffic to the public IP routes through the firewall (routing intent + internet_security_enabled), and Cosmos DB rejects the firewall's PIP with 403 because `public_network_access_enabled = false`. **FIX:** Set `enable_dns_link = true` in Foundry tfvars. This sets VNet DNS to the firewall IP, enabling the centralized DNS architecture: VNet → Firewall DNS proxy → Resolver inbound endpoint → Azure DNS (in dns_vnet context, which HAS zone links) → returns private IP. Confirmed via Microsoft docs that in the centralized DNS pattern, only the resolver VNet needs zone links, not the spoke. The forwarding ruleset "." to 8.8.8.8 linked to dns_vnet is not problematic — Azure-reserved domains (azure.com, windows.net) are exempt from wildcard rules. **SECONDARY RECOMMENDATION:** Change `enable_dns_link` default from `false` to `true` and add a validation block or docs callout, since the module is unusable without it when private DNS is deployed.

- **2025-07-27 (15b-comparison):** Completed line-by-line comparison of Foundry-byoVnet against PG 15b reference (microsoft-foundry/foundry-samples). Root cause of Cosmos DB 403 identified: two security hardening changes from the revamp broke agent proxy networking. (1) **PRIMARY:** `networkAcls.defaultAction` changed from `"Allow"` to `"Deny"` on the Foundry CognitiveServices resource — PG intentionally uses `"Allow"` even with `publicNetworkAccess = "Disabled"` to preserve first-party service access for the agent proxy. Our `"Deny"` blocks the proxy's internal control-plane communication, causing it to use its public IP for Cosmos DB (rejected as 403). (2) **SECONDARY:** `disableLocalAuth` changed from `false` to `true` on the Foundry resource — PG explicitly keeps key auth enabled for the underlying Cognitive Services account; disabling it may force the proxy into a broken auth flow. Additional minor delta: CosmosDB connection category casing (`"CosmosDB"` vs PG's `"CosmosDb"`). Cosmos DB resource itself, storage, private endpoints, RBAC, and capability host are all identical to PG. Written up at `.squad/decisions/inbox/carl-15b-comparison.md`.
- **2025-07-27 (nat-gateway-design):** Produced design document for NAT Gateway as outbound internet alternative when Azure Firewall is off. Key findings: (1) NAT Gateway is supported on spoke VNets in vWAN architecture — confirmed via Microsoft docs. (2) Routing intent overrides NAT Gateway — when firewall is on + routing intent active, spoke NAT Gateways are bypassed. This makes firewall/NAT Gateway mutually exclusive by design. (3) Implicit toggle (`!add_firewall`) chosen over explicit variable to avoid invalid state. (4) NAT Gateway goes in `modules/region-hub/nat-gateway.tf` — 5 resources per region (PIP, NAT GW, PIP association, 2 subnet associations for shared + app subnets). (5) DNS resolver subnets excluded — delegated to Microsoft.Network/dnsResolvers, and the resolver outbound endpoint uses Azure's internal DNS forwarding plane, not subnet-level internet routing. (6) AzureBastionSubnet excluded — unsupported. (7) ALZ modules must deploy their own NAT Gateways — NAT Gateway is VNet-scoped, cannot be shared across VNets. (8) No new user-facing variables needed. (9) Open question: Foundry-managedVnet may not need NAT Gateway at all since Microsoft-managed network has its own outbound rules. Design filed at `docs/nat-gateway-design.md`.
