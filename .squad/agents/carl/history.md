# Project Context

- **Owner:** Ryan Krokson
- **Project:** Azure IaC demo/lab environments — Terraform modules for Azure vWAN, AI Foundry, networking
- **Stack:** Terraform (azurerm >= 4.0, azapi >= 2.0, random ~> 3.5), PowerShell, Azure CLI
- **Structure:** Three root modules — Networking (foundation), Foundry-byoVnet, Foundry-managedVnet linked via terraform_remote_state
- **Created:** 2026-03-27

## Learnings

- **.gitignore audit (2025-01-17):** .gitignore is well-designed for a Terraform demo/lab repo with local state. `.terraform.lock.hcl` files are intentionally tracked (not in .gitignore) — this is correct for reproducible builds in shared repos. Core gap: `.squad/`, `.github/`, and `.copilot/` directories are untracked and should be explicitly ignored. Three Terraform root modules (Networking → Foundry-byoVnet / Foundry-managedVnet) use local backends with state files properly excluded.
- **tfvars pattern:** `.tfvars.example` and `.tfvars.advanced.example` are tracked; actual `.tfvars` are ignored. Pattern correct: `*.tfvars` catches all secrets. No `.example` files are being ignored, which is the right trade-off for IaC docs.
- **2026-03-27 (gitignore-audit finalized):** Completed architecture audit as team lead. Validated with Katia (security), updated by Donut with IDE/env/OS patterns. Decision approved and implemented.
- **2025-07-25 (repo-revamp-architecture-review):** Completed comprehensive review of all .tf files across Networking/, Foundry-byoVnet/, Foundry-managedVnet/ for repo revamp planning. Key findings:
  - Networking main.tf is 1020+ lines monolith; variables.tf has 100+ flat variables with copy-paste per region
  - Zero resource tags anywhere; no outputs from Foundry modules; no outputs.tf in Foundry dirs
  - Cross-module naming inconsistency: `ai_foundry` vs `foundry`, `myrandom` vs `unique`, mixed case in variable names (`firewall_SkuName00`)
  - Provider version drift: Networking uses `>= 4.0, < 5.0`, Foundry modules pin `~> 4.26.0`; Networking missing `required_version`
  - API version drift across azapi resources between Foundry modules
  - Remote state dependency is fragile (hardcoded `../Networking/terraform.tfstate` path); DNS zone IDs constructed as strings from rg_net00_id
  - Region 0/1 code is nearly identical — prime candidate for child module extraction
  - Security gaps: allow-all firewall rules, Key Vault using access policies instead of RBAC, no NSGs, `disableLocalAuth = false` on AI Search
  - Config divergence between Foundry modules: ZRS vs LRS, AzureServices bypass vs no bypass, identity block style differences
  - User preference: Networking = platform landing zone, Foundry modules = optional application landing zones
- **2026-03-29 (repo-revamp-plan):** Full team review completed. Consensus reached on landing zone architecture, implementation priorities, and decision framework. All recommendations merged into squad decisions.md. Next: implementation backlog execution. **Key decision:** Adopt platform/application landing zone model (Networking = platform LZ, Foundry modules = app LZs). Cross-module naming consistency, security hardening, and documentation restructuring are team priorities.
- **2025-07-25 (output-contract-review):** Reviewed Donut's output contract (Networking → Foundry modules). **APPROVED** the Networking outputs — complete, well-described, consistent naming, no sensitive leaks. All 10 DNS zone IDs, 4 subnet IDs, vHub IDs, Key Vault, LAW, and region metadata exported. Foundry modules cleanly consume via `terraform_remote_state` with zero remaining string-interpolated IDs (previous anti-pattern eliminated). DNS zone IDs are still string-constructed in Networking outputs (AVM module constraint), but now centralized in one place instead of scattered. **Flagged:** Foundry module output naming inconsistency — BYO uses `ai_foundry_` prefix, managed uses `foundry_` prefix. Must be standardized before these outputs gain consumers. Filed as follow-up decision.
- **2025-07-25 (tagging-fix):** Fixed Donut's `default_tags` implementation — `default_tags` is an AWS provider concept, not supported by azurerm. Removed invalid blocks from all three config.tf files. Replaced with `local.common_tags` in locals.tf and applied `tags = local.common_tags` to all taggable resources: 36 in Networking, 10 in Foundry-byoVnet, 13 in Foundry-managedVnet. For azapi_resource, used top-level `tags` argument (not inside body) on tracked resources only (skipped child/proxy resources like connections, capabilityHosts, virtualNetworkLinks). All three modules pass `terraform fmt -check` and `terraform validate`. Key learning: azurerm provider does NOT have `default_tags` — always use locals + explicit per-resource tagging.
- **2026-03-30 (tagging-pattern-lock):** Tagged 59 taggable resources across all 3 modules with `local.common_tags` (environment, managed_by, project). Merged tagging strategy decision into decisions.md. Established lasting pattern: azurerm tagging must use locals + explicit per-resource assignment, never default_tags block. Merged inbox decisions (tagging strategy, Foundry output naming standardization, phase1-validation-reject) into decisions.md. Cross-notified Donut of tags pattern. All modules validate successfully.
- **2025-07-26 (local-auth-research):** Researched `disableLocalAuth` settings across all Foundry-connected services. Key findings: (1) AI Search `disableLocalAuth = false` in both modules is NOT required — Foundry agent tool explicitly supports keyless/Entra ID connections and our RBAC roles are already in place. (2) AI Foundry CognitiveServices `disableLocalAuth = false` in byoVnet is inconsistent with managedVnet (which correctly sets `true`) — the old "Azure OpenAI Studio requires key access" caveat applies to the deprecated portal, not the new Foundry portal. (3) Storage and Cosmos DB are already correctly hardened with local auth disabled. Microsoft recommends disabling local auth on all four services for production. The PG sample code defaults were not production-hardened. Written up as `.squad/decisions/inbox/carl-local-auth-research.md` with full citations.
